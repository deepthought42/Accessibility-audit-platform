01

What this is

A short description, for strangers.

Look-see is an open-source platform for finding, explaining, and fixing accessibility issues on real websites. It crawls the pages you ship, walks the journeys your users actually take, and reports on what it sees — in HTML, in rendered pixels, and in the way pages fit together.

Under the hood it is a distributed system: a Neo4j graph representing your site, a fleet of Spring Boot services trading work over Google Cloud Pub/Sub, an Angular interface that watches it in real time, and a growing set of AI-assisted diagnostics that explain why a violation matters, not just that one exists.

It is still being built. That is the fun part.

02

Four accessibility audits, running in parallel

Each one cares about something different. Together they describe a page the way a person would.

Movement I

Content

Alt text that actually describes something. Readability scored with Flesch–Kincaid. Paragraph structure that survives a screen reader's rhythm. The words, checked.

  • img, svg, canvas, iframe, object alt coverage
  • Flesch–Kincaid readability
  • Paragraph density & structure
Movement II

Visual design

Contrast measured from rendered pixels, not from stylesheet rules that may be overridden. Typography above a legibility floor. Whitespace on an 8px grid. Imagery that isn't stolen.

  • WCAG 1.4.3 / 1.4.6 contrast against actual paint
  • 12px minimum, max two typefaces
  • Margin & padding grid validation
  • Image copyright & policy screening
Movement III

Information architecture

Headings in order. Forms with labels. Tables with scope attributes. Links that go somewhere. Languages declared. Reflow that doesn't break. The structural plumbing.

  • Heading hierarchy & H1 discipline
  • Form labelling, fieldsets, required attrs
  • Link validity & colour-only signalling
  • Reflow & text-spacing (1.4.10 / 1.4.12)
Movement IV

Journeys

Not only pages, but the paths through them. A journey-expander discovers interactive elements on the page. A journey-executor drives a real headless browser through them, checking state you only reach by clicking something.

  • Headless Chrome via Selenium WebDriver
  • Dynamic state audited post-interaction
  • Candidate → verified → discarded lifecycle
  • Dead-letter handling for broken flows
03

Why it's unusual

Three choices that put Look-see somewhere other tools aren't.

  1. i.

    It looks at pixels, not only at HTML.

    Colour contrast, font sizing, whitespace and image policy are evaluated against the rendered output of the page. CSS can lie; a screenshot cannot. A page can pass every rule-based checker and still be unreadable for a human — light gray on white, a font that computed to 10px, a button that vanishes below a banner. The visual design service sees what a user sees.

  2. ii.

    It audits journeys, not just URLs.

    Modern applications are state machines. Auditing the landing page is like inspecting a theatre by its lobby. Look-see discovers journeys by scraping interactive elements, executes them in a real browser, and audits the states produced along the way. Discarded paths are recorded; verified ones are re-expanded until the map is complete.

  3. iii.

    It's an engine, not a rule list.

    Beneath the audit services is LookseeCore — models, persistence, browser automation, messaging, cloud, and now LLM. New audits compose from these primitives without re-plumbing infrastructure. The looksee-llm module is the newest: a provider-agnostic layer with Anthropic Claude as the first backend, wired with redaction, caching, budgeting, and retries before any feature ships on it.

04

Inside the machine

The system as it actually runs. No arrows pointing into clouds.

Architectural decisions, with the reasoning intact.

ADR-01 Data model

Use a graph database for a graph-shaped domain.

Accounts own Domains, which own Pages, which contain Elements, which participate in Journeys. Those aren't rows — they are edges. Neo4j lets Look-see query every page where an element appears on a verified journey in a single traversal, instead of five joins. Spring Data Neo4j wires 37 repositories into the core and every service inherits them.

ADR-02 Concurrency

Decouple audit services with Pub/Sub push subscriptions.

The three audit services don't know about each other; they each subscribe to page_audit and run independently. Failures retry. Work scales horizontally by adding Cloud Run instances. The orchestration logic that would otherwise live in an application lives in the broker.

ADR-03 Correctness

Design by Contract, enforced at runtime.

Every service is started with -ea. Preconditions use Objects.requireNonNull and always run; postconditions and invariants use assert and run in test and staging. The contracts are documented, not implied. See LookseeCore/docs/DESIGN_BY_CONTRACT.md.

ADR-04 Platform

Ship on Cloud Run; deploy with Terraform.

Each service is a Docker image. Infrastructure — Pub/Sub topics, Secret Manager entries, Cloud Run services, the Neo4j VM, VPC access — is declared in LookseeIaC/GCP. A fresh environment is one terraform apply.

05

In progress, Spring 2026

Drawn from the commit log, not from a roadmap deck.

  1. April 2026

    Browser module extraction

    com.looksee.browsing became com.looksee.browser. Spring compile-time dependencies were removed. The browser module can now be reused outside the platform — useful for any tool that needs honest, headless page rendering.

  2. March–April 2026

    The Waves refactor

    A five-wave campaign landing in sequenced release branches: observability, a shared base controller, a base Docker image, Spring Boot 3 preparation, an end-to-end pipeline, and a secrets reshuffle. Each wave merges clean.

  3. March 2026

    looksee-llm foundation module

    A provider-agnostic LLM layer joined LookseeCore. Anthropic Claude is the first backend — Opus for full analysis, Haiku for cost-sensitive jobs, Sonnet in the middle. Redaction, Caffeine caching, budgeting, Resilience4j retries and Micrometer metrics are wired before the first feature ships on it. Vision input is supported so audits can reason about screenshots, not only markup.

  4. March 2026

    Trace propagation across Pub/Sub

    Wave 2.2 restored producer-side trace propagation; Wave 5.1 added a smoke driver that walks a single message all the way through the graph. A broken audit is now a distributed trace, not a guess.
